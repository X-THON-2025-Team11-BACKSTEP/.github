## 🌺 Project. 시스투스 (Cistus)
**시스투스(Cistus)**는 산불이 나야만 껍질이 터져 발아하는 '파이로파이트(Pyrophyte)' 식물입니다.
저희는 프로젝트의 **'실패(Fire)'**가 끝이 아니라, 사용자의 성장이 발아하는 **'시작'**이라고 믿습니다.

기존의 성공 중심 포트폴리오 틀을 깨고, 실패 기록을 **데이터 자산(Asset)**으로 치환하여 새로운 기회를 창출합니다.

<br>

## ✨ Key Features

### 1. The Archive: 실패 기록 템플릿
단순한 일기가 아닙니다. 실패를 철저히 분석할 수 있는 **구조화된 회고 템플릿**을 제공합니다.
- **Step-by-Step 회고:** 문제 정의 -> 시도 -> 실패 원인 -> 배운 점
- **실패 태그(Tag):** `기획 변경`, `기술적 한계`, `자금 부족` 등 실패 요인 데이터화

### 2. The Marketplace: 산출물 자산화
버려질 뻔한 프로젝트의 부산물을 **'Unused Asset'**으로 판매하여 실패 비용을 회수합니다.
- **자산 거래:** 디자인 시스템, 기획서, 모듈 코드 등 중간 산출물 판매/공유
- **가치 재발견:** 실패한 프로젝트의 리소스가 누군가에게는 훌륭한 초기 자원이 됩니다.

## 🤝 Ground Rules & Workflow

## 🔗 Git Convention (GitHub Flow 기반)

### 📌 GitHub Flow 개요

> GitHub Flow는 간단하고 효율적인 브랜치 전략으로, 지속적인 배포와 빠른 피드백에 적합합니다.
> 

### 주요 특징

- 단일 **main 브랜치** 사용 (배포 브랜치)
- 모든 개발은 **main** 브랜치에서 파생된 **작업 브랜치**에서 수행
- 브랜치에서 작업 후 **Pull Request(PR)** 생성
- 코드 리뷰 후 **main 브랜치**로 병합
- **지속적인 배포(Continuous Deployment)** 가능

---

## 🌿 Branch Naming Convention

> 형식: 브랜치유형/#이슈번호
> 

### 브랜치 유형

| 유형 | 설명 |
| --- | --- |
| `feat/` | 기능 추가 |
| `fix/` | 버그 수정 |
| `refactor/` | 코드 리팩토링 |
| `test/` | 테스트 코드 작성 |


---

### ✨ Flow

1. **이슈 생성**
2. **작업 브랜치 생성**
    
    > 브랜치명: 브랜치 종류/#이슈번호
    > 
3. 작업 후 **커밋 컨벤션에 맞게 커밋**
4. **Pull Request(PR) 생성**
5. **1명 이상의 코드 리뷰 및 승인**
6. **main 브랜치에 병합**

---

## 🔥 Commit Message Convention

> 형식: 커밋유형: 커밋 메시지 (#이슈번호)
> 

### 커밋 유형

| 유형 | 설명 |
| --- | --- |
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 수정 |
| `style` | 코드 포맷팅, 세미콜론 누락 등 |
| `refactor` | 코드 리팩토링 (기능 변경 없음) |
| `test` | 테스트 코드 추가 및 수정 |
| `chore` | 빌드 설정, 패키지 관리 등 기타 작업 |
| `modify` | 코드 또는 파일 변경 |
| `update` | 최신 상태 유지를 위한 코드 변경 |
